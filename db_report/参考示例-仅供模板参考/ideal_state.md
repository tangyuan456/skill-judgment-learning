# report_skill 学生版理想态

## 1. 能力边界

本 Skill 面向无内部数据库权限的学生环境，只处理本地数据：

- 本地 `.log` / `.xlsx` / `.json` / `.csv` 文件。
- 用户粘贴的标准 records JSON 或原始测试行 JSON。
- 缺少数据源时返回 `missing_data`，停止并要求补充文件。

禁止通过内部 HTTP API、PostgreSQL 或任务标识查询数据。报告中的 TPS/QPS/P95/P99/duration 等数值必须可追溯到本地输入。

## 2. 意图识别

- `single`：默认报告类型，适合单次测试日志。
- `comparison`：命中“对比 / 比较 / vs / 差异 / 哪个更好”。
- `iteration`：命中“迭代 / 版本演进 / 历史趋势 / 变化趋势”。
- `custom`：命中“客制化 / 专项 / 定制 / 深度分析”。

场景词必须做标准化映射，例如“只读 → read_only”“点查 → point_select”“集中式 → 集中式性能”。

## 3. 数据门控

阶段 2 前必须确认：

- 数据源类型、路径和记录数。
- 产品、场景、并发档覆盖范围。
- TPS/QPS/P95 关键字段空值率。
- 缺失字段和不可比条件。

如果输入只有自然语言、没有本地文件或 JSON，不能生成报告。

## 4. 报告质量

- 5 个 sysbench 场景必须独立分析。
- L1/L2/L3 洞察必须分级，L3 只能出现在选型与调优建议章节并标注 `[待确认]`。
- 对比报告必须包含公平性说明。
- 迭代报告必须包含趋势、累计变化和回归点判断。
- md/docx/html 必须独立生成，不能用批量转换替代。

## 5. 红线

- 不编造、不估算、不硬编码性能指标。
- 不跳过三道门控。
- 不把缺数据的任务强行输出成正式报告。
- 不引导学生申请或使用内部数据库查询权限。
