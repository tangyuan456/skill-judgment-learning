# 分析方法论（按 4 类 report_type 分流）

> 本规范定义阶段 3：根据 `intent.report_type` 分发到不同分析模块，输出 `analysis_results.json` + `insights.json`。

---

## 1. 通用约定（4 类报告共享）

### 1.1 标准场景

| 场景 | 主要观测 |
|------|---------|
| `oltp_point_select` | 主键点查，QPS / P95 |
| `oltp_read_only` | 只读事务，TPS / QPS / P95 |
| `oltp_write_only` | 只写事务，TPS / QPS / P95 |
| `oltp_read_write` | 读写混合，TPS / QPS / P95 |
| `oltp_update_index` | 索引更新，TPS / P95 |

### 1.2 洞察分级（强制）

| 等级 | 标识 | 写入位置 |
|------|------|---------|
| L1 | ✅ | 报告任意章节 |
| L2 | 📊 | 报告任意章节 |
| L3 | 💡 | **仅第 5 章选型与调优建议** |

每条 insight 必须包含：
```json
{"level":"L1|L2|L3", "text":"...", "source":"...", "scenario":"oltp_*|all"}
```

### 1.3 公平性核查（强制 + 自动）

`scripts/analyze.py::check_fairness(meta)` 自动校验：
- 数据集一致（`data_config` 相同）
- 并发档对齐（同场景在所有产品上覆盖一致）
- 测试时长一致（`duration_sec`）
- 硬件环境填充

输出 `fairness_warnings: STRONG/WEAK/INFO`。

---

## 2. report_type=single（单次报告）

**输入**：1 份测试数据（1 个 product/version）。

### 2.1 分析维度

| 维度 | 输出 |
|------|------|
| 2.1.1 并发扩展性 | 5 场景每场景 1 条曲线（X=并发，Y=QPS） |
| 2.1.2 P95 延迟稳定性 | 5 场景每场景 1 条曲线（X=并发，Y=P95） |
| 2.1.3 峰值汇总 | 1 张表（5 场景 × 峰值 QPS / 峰值并发 / P95@峰值） |

### 2.2 结论模板

- ✅ L1：`{产品} 在 {场景} 场景，并发 {min}→{max}，QPS {min_qps}→{peak_qps}（{ratio}x），峰值出现在 threads={peak_threads}`
- 📊 L2：当 `peak_threads < max_threads` 且 `qps@max / peak < 0.95` → 输出"过 {peak_threads} 后性能拐点"

---

## 3. report_type=comparison（对比报告）

**输入**：≥ 2 个 product 或 version。

### 3.1 公平性前提（强制）

仅在以下条件全部满足时进行对比：
- 相同场景
- 相同并发档
- 相同数据集（`data_config`）
- 相同硬件等级（差异时显式标注）

任一不满足 → 在 `fairness_warnings.STRONG` 输出，报告中显著标注。

### 3.2 分析维度

| 维度 | 输出（每场景独立）|
|------|------------------|
| 3.2.1 峰值 QPS 对比 | 5 张柱状图（X=产品，Y=峰值QPS） |
| 3.2.2 并发扩展对比 | 5 张多线图（X=并发，Y=QPS） |
| 3.2.3 P95 延迟对比 | 5 张多线图（X=并发，Y=P95） |
| 3.2.4 性能比值矩阵 | 5 场景汇总表（baseline=1.00） |
| 3.2.5 综合评分 | 雷达图（5 维归一化）+ 天梯图 |

### 3.3 结论模板

- ✅ L1：`{场景} 场景：{产品A} 峰值 QPS={X}，{产品B}={Y}，{A 比 B 快/慢 Z%}`
- 📊 L2：`{产品A} 在低并发(<{T})领先，高并发(>{T})被 {产品B} 反超，交叉点 {T}`

---

## 4. report_type=iteration（迭代报告）

**输入**：≥ 3 个时间点/版本（来自同一 product，不同 tool_version）。

### 4.1 分析维度

| 维度 | 计算方法 | 输出 |
|------|---------|------|
| 4.1.1 版本演进趋势 | 每场景每版本峰值 QPS | 5 场景每场景 1 张趋势线（X=版本，Y=峰值QPS） |
| 4.1.2 累计提升 | `(最新峰值 - 首版峰值) / 首版峰值` | 5 场景汇总表 + 1 张柱状图 |
| 4.1.3 回归检测 | 任一版本相对前版本峰值降幅 > `regression_threshold_pct`（默认 5%）| 标记为 ⚠️ 回归点，输出回归列表 |
| 4.1.4 P95 稳定性演进 | 每场景每版本 @ 同并发档的 P95 | 5 张趋势线 |
| 4.1.5 回归原因摘要 | 用户在 intent.other_info 提供（无则跳过）| 文本段 |

### 4.2 结论模板

- ✅ L1：`{场景} 场景：从 {首版} 到 {末版}，峰值 QPS {首值} → {末值}，累计 {±X%}`
- 📊 L2 回归预警：`{场景} 在 {版本} 出现性能回归，相对 {前版本} 峰值下降 {Y%}（>{阈值}%）`
- 📊 L2 持续优化：`{场景} 连续 N 个版本峰值递增，体现持续优化`

### 4.3 必需图表

- 5 场景峰值演进趋势线 × 5
- 累计变化柱状图 × 1（每场景一根柱）
- 回归点散点图 × 1（X=版本，Y=变化率，标红回归点）
- 综合趋势汇总图 × 1（5 场景一张图，归一化值）

## 5. report_type=custom（客制化报告）

**输入**：用户在 `intent.custom_config` 中指定 focus_dimension + extra_questions。

### 5.1 分析维度（按 focus_dimension 分发）

| focus_dimension | 分析方式 |
|----------------|---------|
| `高并发` | 筛选 threads ≥ 用户指定阈值的数据，按场景输出"高并发段 QPS / P95"，对比"中低并发段" |
| `buffer_pool` | 按 `results_key` 中的 buffer_pool 大小分桶，输出 QPS 与 buffer_pool 的散点图 |
| `numa` | 按 `dimension.component` 中 numa 标识分组，对比启用/未启用 |
| `表数量` | 按 `tables=N` 分桶，输出 N 与 QPS 的关系 |
| `表大小` | 按 `table_size` 分桶 |
| `自定义` | 由 extra_questions 引导，分析维度由用户预定义在 `custom_config.dimensions` 中 |

### 5.2 结论模板

按 focus_dimension 模板化，但所有结论必须配套：
- 数据来源（records 子集说明）
- 假设前提（如"假设 buffer_pool 是唯一变量"）
- 不确定性披露（如"该结论基于 N 样本，置信度待加大样本验证"）

### 5.3 必需图表

- 1 张主图：focus_dimension 与 QPS / P95 的关系
- 1 张对照图：focus_dimension 不同档位对比
- 5 场景验证图（如适用）

## 6. analysis_results.json 结构（4 类共享）

```json
{
  "report_type": "single | comparison | iteration | custom",
  "fairness_warnings": [...],
  "single_product": {           // single + comparison 通用
    "<product>": {
      "scale": {"<scenario>":[...]},
      "latency": {"<scenario>":[...]},
      "peak_summary": [...]
    }
  },
  "product_compare": {           // 仅 comparison
    "peak_qps": {"<scenario>":[...]},
    "concurrent_scale": {...},
    "p95_latency": {...},
    "cost_index": {...},
    "radar_scores": {...},
    "overall_score": {...}
  },
  "iteration": {                 // 仅 iteration
    "version_order": ["v1.0", "v1.1", "v2.0"],
    "trend": {"<scenario>": [{"version":"...","peak_qps":...,"delta_pct":...}]},
    "cumulative_change": {"<scenario>": 12.5},
    "regression_points": [{"version":"...","scenario":"...","delta_pct":-7.3}]
  },
  "custom": {                    // 仅 custom
    "focus_dimension": "...",
    "buckets": [...],
    "main_finding": "..."
  }
}
```

## 7. insights.json 结构

```json
{
  "single_product_insights": {
    "<product>": [{"level":"L1","text":"...","source":"...","scenario":"..."}]
  },
  "product_compare_insights": [...],   // comparison
  "iteration_insights": [...],         // iteration
  "custom_insights": [...],            // custom
  "selection_insights": [...]          // 所有类型共享，仅写入第 5 章
}
```

---

## 8. CHECKPOINT③ 断言（强制）

```
ASSERT analysis_results.json 已生成
ASSERT insights.json 已生成
ASSERT 每条 insight 含 level + source 字段
ASSERT L3 不出现在 single_product / product_compare / iteration / custom 四个键中
ASSERT 按 report_type 必备字段都已填写：
  - single:      single_product
  - comparison:  single_product + product_compare
  - iteration:   single_product + iteration
  - custom:      single_product + custom
```

任一断言失败 → 停止流程，输出失败原因。
