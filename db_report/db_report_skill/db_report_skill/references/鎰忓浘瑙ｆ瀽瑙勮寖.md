# 意图解析规范

> 本规范定义阶段 1：从用户自然语言输入提取 `intent.json` 的规则。学生环境只支持本地文件或粘贴 JSON，不支持通过 `task_id` / `plan_id` / `report_id` 查询线上数据库。

---

## 1. intent.json 输出结构

```json
{
  "data_source_type": "local_file | local_data | missing_data",
  "data_source_value": "/path/to/file.log 或 数据 JSON 字符串 或 null",
  "report_type": "single | comparison | iteration | custom",
  "dimension_filters": {
    "test_name_keywords": ["关键词1", "关键词2"],
    "test_name_keywords_or": [["A","B"], ["C"]],
    "tool_name": "sysbench | benchmarksql | null",
    "tool_version_keywords": ["22.6.13", "22.6.14"],
    "component_name": "cvm | proxy | null",
    "component_version": "default | null"
  },
  "results_key_filters": {
    "threads": "1000 | null",
    "tables": "16 | null",
    "table_size": "10000000 | null",
    "time": "300 | null",
    "warehouses": "1000 | null"
  },
  "iteration_config": {
    "version_range": ["v22.6.13", "v22.6.14", "v22.7.0"],
    "regression_threshold_pct": 5
  },
  "custom_config": {
    "focus_dimension": "高并发 / buffer_pool / numa / 其他",
    "extra_questions": ["..."]
  },
  "other_info": "其他补充信息或 null"
}
```

---

## 2. 数据源类型识别

按以下优先级判定：

| 优先级 | 判定规则 | data_source_type |
|--------|---------|------------------|
| 1 | 输入含本地文件路径，且文件存在（`.log` / `.xlsx` / `.json` / `.csv`） | `local_file` |
| 2 | 输入含成对 `{ ... }` 的 JSON 文本，且包含 `meta` / `records` / `test_name` / `results` 等字段 | `local_data` |
| 3 | 无可读取文件、无可解析 JSON | `missing_data` |

示例：

| 输入 | 识别结果 |
|------|---------|
| `读 test-resource/mock_tdsqlb_v22_7_3.log 出报告` | `local_file` |
| `用 test-resource/mock_records_aggregation.json 做对比报告` | `local_file` |
| `{"meta":{},"records":[]}` | `local_data` |
| `生成集中式只读场景报告` | `missing_data` |

识别为 `missing_data` 时必须停止并要求用户提供本地数据文件或粘贴 JSON，不能尝试线上查询或估算数据。

---

## 3. report_type 识别规则

按从上到下优先级，命中即停：

| 关键词正则 | report_type |
|-----------|-------------|
| `对比\|比较\|vs\|差异\|哪个更好` | `comparison` |
| `迭代\|版本演进\|历史趋势\|变化趋势` | `iteration` |
| `客制化\|专项\|定制\|深度分析\|详细分析` | `custom` |
| 其他 | `single` |

迭代关键词命中时，还需要在 `iteration_config.version_range` 中提取 N 个版本/时间点；如果文件中已包含多版本产品，也可在数据确认阶段展示实际版本范围。

---

## 4. 场景关键词中英映射（强制）

用户输入的中文业务术语必须映射为标准场景关键词：

| 中文 | 标准关键词 |
|------|------------|
| 只读 | `read_only` |
| 写入 / 写 | `write_only` 或 `insert` |
| 读写 / 混合读写 | `read_write` |
| 更新索引 | `update_index` |
| 更新非索引 | `update_non_index` |
| 点查 / 点选 | `point_select` |
| 随机点查 | `random_points` |
| 随机范围查询 | `random_ranges` |
| TPCC | `tpmC` 或 `NewOrders` |
| 集中式 | `集中式性能` |
| 分布式 | `分布式性能` |

强制规则：识别到中文术语时，必须把对应英文关键词加入 `test_name_keywords` 数组。

```text
正确："集中式只读" → ["集中式性能", "read_only"]
错误："集中式只读" → ["集中式", "只读"]
```

## 5. AND vs OR 逻辑

### 5.1 何时使用 `test_name_keywords_or`

当用户用「和」「或」「以及」连接不同测试场景时：

| 用户输入 | 正确解析 |
|---------|---------|
| 集中式只读场景和点查场景 | `test_name_keywords_or: [["集中式性能","read_only"],["集中式性能","point_select"]]` |
| 只读或读写测试 | `test_name_keywords_or: [["read_only"],["read_write"]]` |
| insert 以及 update 场景 | `test_name_keywords_or: [["insert"],["update_index"],["update_non_index"]]` |

### 5.2 何时使用 `test_name_keywords`

当用户描述的是同一个场景的多个修饰词：

| 用户输入 | 正确解析 |
|---------|---------|
| 集中式的只读场景 | `test_name_keywords: ["集中式性能","read_only"]` |
| 事务相关的点查测试 | `test_name_keywords: ["事务","point_select"]` |

### 5.3 互斥规则

`test_name_keywords` 和 `test_name_keywords_or` 二选一，不能同时出现。

## 7. 默认值与缺失处理

- 任何字段无法识别 → 设为 `null` 或空数组。
- 必填字段：`data_source_type`, `report_type`。
- `iteration_config` 仅在 `report_type=iteration` 时必填。
- `custom_config` 仅在 `report_type=custom` 时必填。
- `missing_data` 只能进入门控①，不能进入阶段 2 数据接入。
