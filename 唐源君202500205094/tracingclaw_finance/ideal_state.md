# TracingClaw Finance 学生版理想态（评估增强版）

> **对应 Skill**：tracingclaw-finance  
> **版本**：v3.0  
> **设计依据**：Anthropic Agent Eval、HealthBench 多维量规、LLM Agent 评估综述  

## 1. 能力边界

本 Skill 只做金融事实核查，不访问内部链路。可用数据来源只有：

- `westock-data`：结构化金融数据（行情、财报、财务指标、技术指标、股东、分红、评级、板块、指数、宏观等）。
- `mx-finance-search`：公告、研报、财经新闻、政策动态和事件背景。

**强制约束**：
- 用户给内部分享链接时，必须说明学生版无法读取该内容，并请用户粘贴问题与回答后继续验真。
- 两个数据源均无法覆盖时，必须明确返回“当前可用数据不足”，严禁用训练记忆或常识补全具体数字。
- 不提供投资建议、不预测走势。

## 2. 理想流程（对应 Anthropic 的 Transcript 基准）

Skill 必须严格遵循以下步骤（过程评估 10% 权重，用于诊断路径偏差）：

1. **识别输入类型**：判断是纯问题、问题+候选回答、还是内部分享链接请求。
2. **拆解金融意图**：将问题拆成最小可核验单元，明确涉及的市场、标的、时间、指标类型。
3. **优先调用结构化数据**：先用 `westock-data` 查询行情、财报、财务指标等结构化事实。
4. **辅助调用资讯数据**：用 `mx-finance-search` 补充公告、研报、新闻和事件背景。
5. **口径对齐**：逐项标注市场、币种、财年/自然年、交易日、复权方式等口径。
6. **对照候选回答**：按覆盖度、正确性、时效性、口径一致性逐项判断。
7. **输出结构化结论**：含 0/1/2 分真实性评分、核查明细、口径与风险说明、修正参考答案。

> **方法论注**：此为 Skill 的理想执行轨迹，用于检测“是否跳过关键步骤”。但主评估（90%权重）仍基于最终产出——遵循 Anthropic“评估终点而非过程”原则。

## 3. 质量标准（HealthBench 多维量规的具象化）

### 3.1 硬性合规规则（L1/L2 Grader 可自动检查）
- 每个关键数字必须有来源（指定 Skill 名和字段）。
- 财报必须标注报告期、币种和财年/自然年口径。
- 行情必须标注交易日或更新时间。
- 新闻/研报观点不能替代结构化财务事实，必须注明“观点性内容”。
- 无法查询时必须说明数据缺口，不能补写数字。
- 货币单位必须正确：港股港元/美元、美股美元、日股日元，不得混用人民币。

### 3.2 可量化的质量维度（用于 LLM Judge 打分，权重按结果评估）

| 维度（权重） | 评分标准（锚点） |
|-------------|----------------|
| **评分准确性 (40%)** | 0 分：真实性评分与 ground truth 完全不符；1 分：主评分正确但有边界偏差；2 分：完全正确，判断理由与 ground truth 对齐 |
| **证据链完整性 (30%)** | 0 分：无证据或捏造；1 分：来源模糊；2 分：每条关键判断有明确来源和字段，但可能缺反例；3 分：证据链完整，结构化数据优先，冲突时有优先级说明 |
| **口径严谨性 (20%)** | 0 分：口径混淆；1 分：部分标注但有遗漏；2 分：所有涉及的市场、币种、财年、交易日等均显式标注 |
| **边界遵守 (10%)** | 触发红线即 0 分；否则满分，并检查数据不足时的声明行为 |

> 加权总分：Score = (评分准确性/2×40) + (证据链/3×30) + (口径/2×20) + (边界/2×10)。通过线 ≥ 80 且评分准确性不得为 0。

## 4. 红线

- 不调用内部链路接口。
- 不要求学生提供内部 Token、日志或内部权限。
- 不把内部分享链接当作可直接读取的数据源。
- 不编造股价、财报、公告或评级。
- 不混用 A 股、港股、美股市场口径（除非已显式标注并说明合理性）。
- 不把新闻/研报观点伪装成客观事实。

## 附录 A：Grader 分层策略（与方法论映射）

| 层级 | 内容 | 实现 | 对应概念 |
|------|------|------|---------|
| L1 | 结构完整性、红线关键词 | 代码正则/JSON Schema | Grader（确定性） |
| L2 | 证据引用格式、口径标注存在性 | 正则提取 | Grader（半自动） |
| L3 | 语义质量（评分、证据、口径） | LLM-as-a-judge + 上表量规 | HealthBench 量规 |
| L4 | 争议校准 | 人工抽检（Kappa ≥0.8） | Anthropic 校准 |

## 附录 B：Trial 与 Meta 评估摘要

- **Trial**：每条用例执行 3 次，至少 2 次评分一致且总分 ≥80 且无红线方可通过。
- **Meta 覆盖**：须确保用例集覆盖 0/1/2 分场景、全部红线类型、数据不足和内部分享拒绝场景，并通过判定一致性验证（Kappa ≥0.8）。

## 附录 C：当前版本的已知局限与迭代方向

- 维度权重需基于大规模评测数据调优。
