# 意图解析规范

> 本规范定义阶段 1：从用户自然语言输入提取出 `intent.json` 的完整规则。
> 大量内容直接源自 `dify yml` 中「客户需求拆解」节点的 system prompt，并增补「数据源类型识别」与「场景中英映射」两块。

---

```json
{
  "data_source_type": "local_file | local_data | task_id | keyword_only",

  "report_type": "single | comparison | iteration | custom",
  "dimension_filters": {
    "test_name_keywords": ["关键词1", "关键词2"],
    "test_name_keywords_or": [["A","B"], ["C"]],

    "tool_version_keywords": ["22.6.13", "22.6.14"],
```

## 2. 数据源类型识别（新增，本 Skill 独有）

按以下优先级判定：

| 优先级 | 判定规则 | data_source_type |
|--------|---------|------------------|
| 1 | 输入含本地文件路径，且文件存在（`.log` / `.xlsx` / `.json` / `.csv`） | `local_file` |
| 2 | 输入含成对 `{ ... }` 的 JSON 文本，且包含 `test_name` / `results` 等字段 | `local_data` |
| 3 | 输入含 `task_id` / `plan_id` / `report_id` 关键词，或匹配 `^[a-zA-Z0-9_]{8,64}$` 且包含 `_tdsql_` / `_intel_x86` / `_kunpeng_arm` 等典型片段 | `task_id` |
| 4 | 仅含场景关键词（无标识符无文件） | `keyword_only` |

**示例**：

| 输入 | 识别结果 |
|------|---------|
| `读 /Users/muz/data/test.log 出报告` | `local_file` |
| `输出 task_id 为 878fd4d4_tdsql_intel_x86 的报告` | `task_id` |
| `dd270144` | `task_id`（纯标识符） |
| `查询集中式只读场景` | `keyword_only` |
| `{"test_name":"集中式_read_only","results":{"qps":50000}}` | `local_data` |

---

## 3. report_type 识别规则

按从上到下优先级，命中即停：

| 关键词正则 | report_type |
|-----------|-------------|
| `对比\|比较\|vs\|差异\|哪个更好` | `comparison` |
| `迭代\|版本演进\|历史趋势\|变化趋势` | `iteration` |
| `客制化\|专项\|定制\|深度分析\|详细分析` | `custom` |
| 其他 | `single` |

⚠️ 注意：迭代关键词命中时还需要在 `iteration_config.version_range` 中提取 N 个版本/时间点。

---

## 4. 场景关键词中英映射（强制）

用户输入的中文业务术语必须映射为数据库英文字段：

| 中文 | 数据库字段（test_name 中包含） |
|------|------------------------------|
| 只读 | `read_only` |
| 写入 / 写 | `write_only` 或 `insert` |
| 读写 / 混合读写 | `read_write` |
| 更新索引 | `update_index` |
| 更新非索引 | `update_non_index` |
| 点查 / 点选 | `point_select` |
| 随机点查 | `random_points` |
| 随机范围查询 | `random_ranges` |
| TPCC | `tpmC` 或 `NewOrders` |
| 集中式 | `集中式性能` |
| 分布式 | `分布式性能` |

**强制规则**：识别到中文术语时，**必须把对应英文关键词加入 `test_name_keywords` 数组**。

```
✅ "集中式只读"   → ["集中式", "read_only"]
❌ "集中式只读"   → ["集中式", "只读"]
```

## 5. AND vs OR 逻辑（最易错！）

### 5.1 何时使用 `test_name_keywords_or`（OR 逻辑）

当用户用 **「和」「或」「以及」**连接 **不同测试场景** 时：

| 用户输入 | 正确解析 |
|---------|---------|
| 集中式只读场景 **和** 点查场景 | `test_name_keywords_or: [["集中式","read_only"],["集中式","point_select"]]` |
| 只读 **或** 读写测试 | `test_name_keywords_or: [["read_only"],["read_write"]]` |
| insert **以及** update 场景 | `test_name_keywords_or: [["insert"],["update_index"],["update_non_index"]]` |

### 5.2 何时使用 `test_name_keywords`（AND 逻辑）

当用户描述的是 **同一个场景** 的多个修饰词：

| 用户输入 | 正确解析 |
|---------|---------|
| 集中式的只读场景 | `test_name_keywords: ["集中式","read_only"]` |
| 事务相关的点查测试 | `test_name_keywords: ["事务","point_select"]` |

### 5.3 互斥规则

`test_name_keywords` 和 `test_name_keywords_or` **二选一**，不能同时出现。

---

## 6. task_id 与场景关键词共存规则（最易错）

❌ **错误做法**：看到 task_id 就把 `test_name_keywords` 设为 `[]`
✅ **正确做法**：仔细检查用户是否提到了场景词，**两者必须共存**

| 用户输入 | task_id | test_name_keywords |
|---------|---------|---------------------|
| 查询任务 dd270144 的所有测试结果 | `dd270144` | `[]` |
| 输出 task_id 为 878fd4d4 的 **集中式** 场景报告 | `878fd4d4` | `["集中式"]` |
| 输出 task_id 为 878fd4d4 的 **集中式只读** 场景报告 | `878fd4d4` | `["集中式","read_only"]` |
| 集中式场景报告 | `null` | `["集中式"]` |

## 8. 默认值与缺失处理

- 任何字段无法识别 → 设为 `null` 或空数组
- 必填字段：`data_source_type`, `report_type`
- `iteration_config` 仅在 `report_type=iteration` 时必填
- `custom_config` 仅在 `report_type=custom` 时必填
