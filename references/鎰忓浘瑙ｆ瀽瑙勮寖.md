# 意图解析规范

> 本规范定义阶段 1：从用户自然语言输入提取出 `intent.json` 的完整规则。

---

## 1. intent.json 输出结构

```json
{
  "data_source_type": "local_file | local_data | keyword_only",
  "data_source_value": "/path/to/file.log 或 数据 JSON 字符串",
  "report_type": "single | comparison | iteration | custom",
  "dimension_filters": {
    "test_name_keywords": ["关键词1", "关键词2"],
    "test_name_keywords_or": [["A","B"], ["C"]],
    "tool_name": "sysbench | benchmarksql | null",
    "tool_version_keywords": ["v1.0", "v1.1"],
    "component_name": "cvm | proxy | null",
    "component_version": "default | null"
  },
  "results_key_filters": {
    "threads": "1000 | null",
    "tables": "16 | null",
    "table_size": "10000000 | null",
    "time": "300 | null",
    "warehouses": "1000 | null"
  },
  "task_filters": {
    "task_id": null,
    "plan_id": null,
    "report_id": null
  },
  "iteration_config": {
    "version_range": ["v1.0", "v1.1", "v2.0"],
    "regression_threshold_pct": 5
  },
  "custom_config": {
    "focus_dimension": "高并发 / buffer_pool / numa / 其他",
    "extra_questions": ["..."]
  },
  "other_info": "其他补充信息或 null"
}
```

---

## 2. 数据源类型识别

按以下优先级判定：

| 优先级 | 判定规则 | data_source_type |
|--------|---------|------------------|
| 1 | 输入含本地文件路径，且文件存在（.log / .xlsx / .json / .csv） | `local_file` |
| 2 | 输入含成对 `{ ... }` 的 JSON 文本，且包含 `product` / `scenario` / `tps` 等字段 | `local_data` |
| 3 | 仅含场景关键词（无文件） | `keyword_only` |

**示例**：

| 输入 | 识别结果 |
|------|---------|
| `读 /Users/muz/data/test.log 出报告` | `local_file` |
| `对比 test.log 和 test2.log` | `local_file` |
| `查询集中式只读场景` | `keyword_only` |
| `{"product":"test-db","scenario":"oltp_read_only","tps":12345}` | `local_data` |

---

## 3. report_type 识别规则

按从上到下优先级，命中即停：

| 关键词正则 | report_type |
|-----------|-------------|
| `对比\|比较\|vs\|差异\|哪个更好` | `comparison` |
| `迭代\|版本演进\|历史趋势\|变化趋势` | `iteration` |
| `客制化\|专项\|定制\|深度分析\|详细分析` | `custom` |
| 其他 | `single` |

---

## 4. 场景关键词中英映射（强制）

用户输入的中文业务术语必须映射为英文字段：

| 中文 | 英文关键词 |
|------|------------------------------|
| 只读 | `read_only` |
| 写入 / 写 | `write_only` |
| 读写 / 混合读写 | `read_write` |
| 更新索引 | `update_index` |
| 更新非索引 | `update_non_index` |
| 点查 / 点选 | `point_select` |
| 随机点查 | `random_points` |
| 随机范围查询 | `random_ranges` |
| TPCC | `tpmC` |
| 集中式 | `集中式性能` |
| 分布式 | `分布式性能` |

---

## 5. AND vs OR 逻辑（最易错！）

### 5.1 何时使用 `test_name_keywords_or`（OR 逻辑）

当用户用 **「和」「或」「以及」**连接 **不同测试场景** 时：

| 用户输入 | 正确解析 |
|---------|---------|
| 集中式只读场景 **和** 点查场景 | `test_name_keywords_or: [["集中式","read_only"],["集中式","point_select"]]` |
| 只读 **或** 读写测试 | `test_name_keywords_or: [["read_only"],["read_write"]]` |

### 5.2 何时使用 `test_name_keywords`（AND 逻辑）

当用户描述的是 **同一个场景** 的多个修饰词：

| 用户输入 | 正确解析 |
|---------|---------|
| 集中式的只读场景 | `test_name_keywords: ["集中式","read_only"]` |

### 5.3 互斥规则

`test_name_keywords` 和 `test_name_keywords_or` **二选一**，不能同时出现。

## 6. 提取示例（端到端）

### 示例 1：本地文件 + 默认单次报告

**输入**：`生成 /Users/muz/data/sysbench.log 的测试报告`

```json
{
  "data_source_type": "local_file",
  "data_source_value": "/Users/muz/data/sysbench.log",
  "report_type": "single",
  "dimension_filters": {"test_name_keywords":[], "test_name_keywords_or":null, "tool_name":null, "tool_version_keywords":[], "component_name":null, "component_version":null},
  "results_key_filters": {"threads":null, "tables":null, "table_size":null, "time":null, "warehouses":null},
  "task_filters": {"task_id":null, "plan_id":null, "report_id":null},
  "iteration_config": null,
  "custom_config": null,
  "other_info": null
}
```

### 示例 2：本地文件 + 对比报告

**输入**：`对比 test.log 和 test2.log 的集中式只读性能`

```json
{
  "data_source_type": "local_file",
  "data_source_value": "test.log",
  "report_type": "comparison",
  "dimension_filters": {"test_name_keywords":["集中式","read_only"], "test_name_keywords_or":null, "tool_name":null, "tool_version_keywords":[], "component_name":null, "component_version":null},
  "task_filters": {"task_id":null, "plan_id":null, "report_id":null},
  "other_info": "对比两个日志文件在集中式只读场景下的性能"
}
```

### 示例 3：迭代报告

**输入**：`生成 v1.0、v1.1、v2.0 三个版本的迭代演进报告`

```json
{
  "data_source_type": "keyword_only",
  "data_source_value": null,
  "report_type": "iteration",
  "dimension_filters": {"test_name_keywords":[], "tool_version_keywords":["v1.0","v1.1","v2.0"]},
  "iteration_config": {
    "version_range": ["v1.0","v1.1","v2.0"],
    "regression_threshold_pct": 5
  },
  "other_info": "三版本迭代演进，回归阈值 5%"
}
```

### 示例 4：客制化深度分析

**输入**：`生成客制化报告，深度分析高并发下 buffer pool 大小对 QPS 的影响`

```json
{
  "data_source_type": "keyword_only",
  "report_type": "custom",
  "custom_config": {
    "focus_dimension": "buffer_pool",
    "extra_questions": ["高并发下 buffer pool 大小对 QPS 的影响"]
  },
  "other_info": "客制化深度分析：高并发 + buffer pool 维度"
}
```

---

## 7. 默认值与缺失处理

- 任何字段无法识别 → 设为 `null` 或空数组
- 必填字段：`data_source_type`, `report_type`
- `iteration_config` 仅在 `report_type=iteration` 时必填
- `custom_config` 仅在 `report_type=custom` 时必填
